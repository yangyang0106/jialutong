# 家路通架构收敛路线图

## 背景

家路通已经从微信小程序 MVP 演进到具备路线创建、AI 建议、审核发布、TTS、模拟执行和行程复盘的产品雏形。

当前主要风险不再是功能不足，而是快速迭代留下的结构债：

1. 历史项目与家路通混在同一工作区。
2. 路线数据仍以 JSON 文件为主。
3. 后端入口文件承担过多配置、外部服务和依赖装配职责。
4. 前端仍保留较多路线生成逻辑。
5. 文档以进度流水账为主，缺少稳定架构入口。

## 下一阶段路线

### P0：迁数据

目标：从 JSON 文件路线存储迁移到结构化数据库。

优先迁移：

1. `routes`
2. `route_steps`
3. `trip_results`
4. `uploaded_assets`
5. `emergency_contacts`

迁移原则：

- 先双写或导入，不一次性删除 JSON。
- 保留回滚路径。
- 发布路线、行程结果和求助记录必须可审计。
- `familyId` 和 `elderId` 必须成为数据库查询隔离条件。

### P0：求助联系人服务端化

目标：求助联系人不再只存在小程序本地 storage。

原因：

- 换手机后本地配置会丢失。
- 老人端与家属端无法共享最新联系人。
- 未来多个求助联系人和优先级必须依赖服务端。

第一阶段只支持单个默认求助联系人。

### P1：拆后端

当前 `main.py` 仍承担过多职责。

拆分目标：

```text
app/
├── core/
│   ├── config.py
│   └── security.py
├── dependencies.py
├── integrations/
│   ├── baidu_map.py
│   ├── wechat.py
│   ├── tencent_tts.py
│   └── llm.py
├── repositories/
│   ├── route_repository.py
│   ├── trip_result_repository.py
│   └── contact_repository.py
└── services/
```

原则：

- `main.py` 只负责创建 FastAPI app 和 include router。
- 外部服务调用集中到 `integrations`。
- 数据读写集中到 `repositories`。
- 业务规则集中到 `services`。

### P1：收前端路线

目标：路线生成、决策点提取、默认语音生成逐步后移到后端。

小程序保留：

1. 路线创建表单
2. 审核展示和编辑
3. 老人端路线执行
4. 离线缓存和固定兜底路线

小程序逐步移除：

1. 百度原始路线解析
2. 决策点提取
3. 路线草稿生成核心规则

## 当前不做

- 不在目录迁移同一批次里直接改数据库存储。
- 不在同一批次里重写路线引擎。
- 不在同一批次里拆完全部后端。
- 不引入复杂多角色平台。

原因：避免同时改变目录、数据、接口和运行逻辑，降低回归风险。
