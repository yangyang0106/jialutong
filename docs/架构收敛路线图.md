# 家路通架构收敛路线图

## 背景

家路通已经从微信小程序 MVP 演进到具备路线创建、AI 建议、审核发布、TTS、模拟执行和行程复盘的产品雏形。

当前主要风险不再是功能不足，而是快速迭代留下的结构债：

1. 历史项目与家路通混在同一工作区。
2. 路线数据仍以 JSON 文件为主。
3. 后端入口文件承担过多配置、外部服务和依赖装配职责。
4. 前端仍保留较多路线生成逻辑。
5. 文档以进度流水账为主，缺少稳定架构入口。

## 下一阶段路线

### P0：生产止血

目标：先消除会破坏产品通用性和生产数据安全的明显风险。

当前状态：已完成第一轮。

已落地：

1. 老人端 `pages/route/route.js` 删除针对 `14 号线`、`887 路`、`临洮路` 等特定路线的硬编码短任务判断，优先使用服务端返回的 `elderShortAction` / `shortAction`。
2. 小程序生产包默认不再内置真实演示路线，`data/routes.js` 为空；远程不可用时只使用上次缓存。离线演示必须在本地 `config/upload.local.js` 显式打开 `enableLocalDemoRoutes`。
3. 后端管理员权限判断统一到 `app.auth.require_family_admin`，各路由不再各自维护一份 `_require_family_admin`。
4. 部署文档明确当前 JSON 路线数据只支持单 uvicorn worker，迁数据库前禁止多 worker 或多 api 容器。

剩余风险：

1. 路线、步骤和行程结果仍在 JSON 文件中，单 worker 只是止血，不是长期方案。
2. 小程序前端仍保留部分路线引擎兼容代码，后续需要继续收敛。

### P0：迁数据

目标：从 JSON 文件路线存储迁移到结构化数据库。

优先迁移：

1. `routes`
2. `route_steps`
3. `trip_results`
4. `uploaded_assets`
5. `emergency_contacts`

迁移原则：

- 先双写或导入，不一次性删除 JSON。
- 保留回滚路径。
- 发布路线、行程结果和求助记录必须可审计。
- `familyId` 和 `elderId` 必须成为数据库查询隔离条件。

当前状态：已完成迁移前的仓储抽象。

已落地：

1. 新增 `app/repositories/json_repository.py`，统一封装路线配置、路线草稿和行程结果 JSON 读写。
2. `main.py` 不再直接读写 JSON 文件路径，而是通过 repository 接口访问数据。
3. 后续迁 SQLite/PostgreSQL 时，应替换 repository 实现，路由层和业务服务不再直接感知文件路径。

尚未完成：

1. 路线、步骤、行程结果仍存储在 JSON 文件中。
2. 尚未实现 JSON → 数据库导入脚本。
3. 尚未实现双写或回滚策略。

### P0：求助联系人服务端化

目标：求助联系人不再只存在小程序本地 storage。

原因：

- 换手机后本地配置会丢失。
- 老人端与家属端无法共享最新联系人。
- 未来多个求助联系人和优先级必须依赖服务端。

第一阶段只支持单个默认求助联系人。

当前状态：已完成第一阶段。

已落地：

1. 服务端新增按家庭和老人档案隔离的默认求助联系人读写能力。
2. 家属端保存联系人时同步到账号服务。
3. 老人端读取路线和求助联系人时优先使用服务端家庭数据。
4. 未登录或网络不可用时，小程序继续保留本机配置作为演示和离线兜底。
5. 普通老人账号只能读取联系人，不能修改家庭求助联系人。

后续增强：

1. 多求助联系人和优先级。
2. 求助事件通知与处理状态闭环。
3. 联系人变更审计。
4. 与行程 HELP 结果联动复盘。

### P1：拆后端

当前 `main.py` 仍承担过多职责。

拆分目标：

```text
app/
├── core/
│   ├── config.py
│   └── security.py
├── dependencies.py
├── integrations/
│   ├── baidu_map.py
│   ├── wechat.py
│   ├── tencent_tts.py
│   └── llm.py
├── repositories/
│   ├── route_repository.py
│   ├── trip_result_repository.py
│   └── contact_repository.py
└── services/
```

原则：

- `main.py` 只负责创建 FastAPI app 和 include router。
- 外部服务调用集中到 `integrations`。
- 数据读写集中到 `repositories`。
- 业务规则集中到 `services`。

当前状态：已完成 P1 主体收敛。

已落地：

1. 新增 `app/core/config.py`，集中加载环境变量、路径和外部服务配置。
2. 新增 `app/repositories/json_repository.py`，集中处理 MVP 阶段 JSON 存储。
3. 新增 `app/repositories/protocols.py`，定义 `RouteConfigRepository`、`EngineRouteRepository`、`TripResultRepository` 和 `AppRepository` 协议，为 SQLite/PostgreSQL 替换 JSON 存储预留稳定接口。
4. 新增 `app/dependencies.py` 的 `AppContainer`，统一装配 settings、repository、auth store、锁、外部服务调用和 router 依赖。
5. `main.py` 已收敛为薄入口，只负责加载配置、创建 `FastAPI`、挂载 `/files` 和注册路由。
6. `auth.py` 的建表 SQL 已抽离到 `app/auth_schema.py`，账号业务类不再直接内嵌大段 schema 文本。
7. `AppSettings.ssl_context` 改为缓存属性，避免每次外部请求重复创建 SSLContext。

尚未完成：

1. 外部服务调用目前集中在 `AppContainer`，尚未进一步拆成独立 `integrations/` 目录。
2. `auth.py` 仍然偏大，后续可以继续拆成 `auth_repository`、`auth_service`、`auth_dto` 和正式迁移脚本。
3. 路由层仍接收函数参数，但来源已统一为 `AppContainer`，下一步可按领域服务对象继续收束。

### P1：收前端路线

目标：路线生成、决策点提取、默认语音生成逐步后移到后端。

小程序保留：

1. 路线创建表单
2. 审核展示和编辑
3. 老人端路线执行
4. 离线缓存和固定兜底路线

小程序逐步移除：

1. 百度原始路线解析
2. 决策点提取
3. 路线草稿生成核心规则

当前状态：已完成 P1 主体收敛。

已落地：

1. 小程序前端 `utils/route-engine/` 兼容实现已删除；百度原始路线解析、决策点提取和 RouteStep 生成统一由后端承担。
2. 新功能不得继续扩大小程序端路线生成规则。
3. 服务端新增 `app/services/route_engine/`，承接百度路线标准化、决策点提取、默认语音生成和 RouteStep 生成。
4. 服务端新增 `POST /api/engine/routes/from-baidu`，小程序正式创建路线草稿时调用该接口。
5. 小程序 `createAndSaveRouteDraft` 已改为提交百度原始路线给服务端生成草稿，不再本地生成正式草稿。
6. 服务端新增 `POST /api/engine/routes/plan-summaries`，AI 路线顾问前的候选路线摘要也由后端根据百度原始路线生成。
7. 小程序 `route-service.js` 不再直接引入 `baidu-route-parser.js` 和 `decision-point-extractor.js`，正式创建/建议链路都通过服务端路线引擎。

尚未完成：

1. 小程序端兼容版路线引擎已删除；单测与模拟演示改为通过后端 API 和运行时 adapter 验证。
2. `route-builder.js`、`baidu-route-parser.js`、`decision-point-extractor.js` 仍被前端单测覆盖，后续确认无历史兼容需求后再删除。
3. 前后端路线引擎在一段时间内需要双轨测试，确认真实百度响应下结果一致后，再删除小程序端兼容规则。

## 当前不做

- 不在目录迁移同一批次里直接改数据库存储。
- 不在同一批次里重写路线引擎。
- 不在同一批次里拆完全部后端。
- 不引入复杂多角色平台。

原因：避免同时改变目录、数据、接口和运行逻辑，降低回归风险。
