# 家路通产品进度整理

更新日期：2026-06-24

## 1. 当前定位

家路通不是地图软件，不是导航软件，也不是 AI 聊天软件。

家路通的长期产品定位是：

```text
帮助老人安全出行的家庭协同系统。

老人表达需求
AI 处理复杂信息
家庭成员安全把关
路线承载一次可复用的安心出行
```

最高原则：

```text
老人负责表达需求，而不是负责操作系统。
```

老人可以说“我要去医院”“我要去妈妈家”“我要去公园”“我要回家”，但不应该承担输入地址、搜索地点、比较路线、审核路线、配置锚点、上传照片、录制语音等复杂操作。这些工作应由 AI 或家庭成员承担。

老人端不依赖读字、不依赖看地图、不让老人选择路线方案。地图服务、大模型和 TTS 都只服务于家庭成员创建和审核路线，以及后台判断老人是否接近锚点、是否偏离路线。

### 1.1 产品参与者

```text
Elder      老人，负责表达需求、使用路线、反馈问题和发起求助。
Guardian   守护人，负责审核路线、补充照片、补充地标、补充语音和处理求助。
Member     家庭成员，负责协助创建和维护路线。
AI         路线顾问和内容助手，只能生成建议，不能审核或发布路线。
Route      家庭共同维护的安全出行资产。
```

Guardian 是安全责任人。AI 不是审核人，不是发布人，永远不能自动发布路线。

### 1.2 产品工作流

```text
老人表达需求
→ AI 生成路线草稿
→ 家庭成员审核与完善
→ Guardian 发布路线
→ 老人按语音和照片使用路线
→ 记录 FOUND / NOT_FOUND / HELP
→ 家庭成员和 AI 持续优化路线
```

任何家庭成员都可以创建路线草稿。创建权、审核权、发布权必须分离。

### 1.3 独居老人边界

必须支持独居老人场景。如果老人没有 Guardian，允许老人创建、审核和发布自己的路线，但系统必须明确标记：

```text
未经过守护人审核
```

该状态不阻止使用，但要降低安全承诺，并强化求助和复盘。

### 1.4 后续功能评审准则

未来新增功能必须同时满足：

- 老人只负责表达需求或做极少量确认。
- AI 负责整理路线、文案、风险和采集建议。
- 家庭成员负责安全把关。
- 重要操作不能依赖老人阅读大量文字。
- 语音优先，照片辅助，地图和 GPS 只做后台判断。

如果某个功能要求老人完成复杂配置、判断路线优劣、理解技术信息或承担安全审核责任，必须重新评审。

## 2. 当前阶段结论

当前已经完成从路线创建到路线执行的 MVP 闭环：

```text
家属输入起点和终点
→ 百度地图生成候选路线
→ 路线顾问给出建议
→ 系统提取决策锚点
→ AI 生成五类老人语音文案
→ 批量生成 TTS
→ AI 生成真实路线采集清单
→ 家属补照片、审核、发布
→ 老人端按语音和照片执行
→ GPS 后台触发接近、到达、偏航
→ 记录 FOUND / NOT_FOUND / HELP
```

当前适合继续做：

- 家属端路线创建演示
- 步行、公交路线真实数据验证
- AI 语音建议演示
- TTS 批量生成演示
- AI 真实路线采集清单演示
- 开发模式模拟路线执行
- 第一条真实路线的家属陪同实地测试

当前不适合直接做：

- 老人独立真实出行
- 大规模公开发布
- 复杂地铁站厅导航
- 无家属审核的自动发布
- 无 HTTPS 公网服务的真机正式使用

## 3. 已完成能力

### 3.0 架构治理

已完成：

- 家路通和 A 股助手已拆成两个独立项目目录，避免上下文和依赖继续混用。
- 家路通后端新增集中配置模块 `app/core/config.py`。
- 家路通后端新增 JSON 仓储抽象 `app/repositories/json_repository.py`。
- `main.py` 不再直接散落管理 JSON 文件路径，后续迁数据库可以替换 repository 实现。
- 家路通后端新增 `app/services/route_engine/`，负责把百度原始路线转换为家路通 RouteStep。
- 小程序创建路线草稿已改为调用服务端 `POST /api/engine/routes/from-baidu`。
- 小程序路线引擎已降级为兼容层，只用于本地单测、离线演示和历史草稿兼容。

当前仍需收敛：

- 路线、步骤和行程结果仍存储在 JSON 文件中。
- 路线候选摘要仍在小程序端生成，后续可继续后移。
- 百度地图、微信、TTS、LLM 调用仍需进一步拆到 `integrations` 层。

### 3.1 老人端

已完成：

- 白色、暖橙色适老化首页。
- 首页只保留家属发布的自定义路线按钮、紧急求助和弱化的家属配置入口。
- 首页路线按钮不再固定为“去妈妈家 / 回家”，按钮文字使用家属创建时填写的路线名称。
- 导航页以语音、照片、大按钮为主。
- 去除大面积深绿色、技术信息和复杂地图。
- 不显示 GPS 误差、精度、错误码等程序员信息。
- 照片区域为视觉主体。
- 底部固定按钮：
  - 再听一遍
  - 我找到了，下一步
  - 长按 3 秒求助
- 求助长按触发，避免误触。
- 到达锚点后不自动切换步骤，需要老人或陪同者确认。
- 偏航时提示停下并进入求助状态。
- 本机保存行程进度，可恢复。
- 老人端求助号码优先读取家庭账号下的服务端联系人；未登录或网络不可用时，仍使用本机配置兜底。

### 3.2 路线创建

已完成：

- 家属可创建新路线。
- 新建路线支持自定义路线名称，例如“去医院”“去菜场”“去女儿家”。
- 创建时选择的是“首页按钮 1 / 首页按钮 2”的显示位置，不再把路线名称写死为“去妈妈家 / 回家”。
- 支持起点、终点搜索。
- 支持当前位置作为起点或终点。
- 使用当前位置时支持百度逆地理解析。
- 支持步行和公交优先。
- 百度地图服务端 AK 不进入小程序代码包。
- 百度原始路线转换为家路通 RouteStep 的主路径已迁到服务端。
- 路线草稿自动保存。
- 草稿支持打开和删除。
- 已发布路线禁止删除。
- 首页优先读取对应按钮的已发布路线。

### 3.2.1 账号与求助联系人

已完成：

- 微信登录作为家庭账号入口，服务端只保存 openid 和家路通 session token。
- 小程序端不保存微信 `session_key`，不再依赖旧上传 token 作为长期登录凭证。
- 家庭账号作为当前 MVP 的数据隔离边界。
- FAMILY_ADMIN 作为当前阶段 Guardian，可创建、审核、发布路线和配置求助联系人。
- FAMILY_MEMBER 不能越权发布路线。
- 默认老人档案可绑定到老人微信账号，老人端读取对应家庭下已发布路线。
- 求助联系人已服务端化，支持按家庭和老人档案保存默认联系人：
  - 联系人姓名
  - 与老人的关系
  - 求助电话
- 家属在“家路通配置 → 求助联系人”里保存后，会同步到服务端；未登录时仍保存在本机，便于演示和离线兜底。
- 老人端读取求助联系人时按当前家庭和老人档案隔离，避免不同家庭串用号码。

当前边界：

- 本阶段只支持单个默认求助联系人。
- 多联系人、优先级、备用联系人、通知闭环后置。
- 生产发布前必须填写真实联系人，不能使用测试号码。

### 3.3 路线解析与锚点生成

已完成：

- 百度步行路线解析。
- 百度公交路线解析。
- 公交前后步行接驳段二次展开。
- 基础地铁路线解析。
- 站内换乘识别为 `TRANSFER`。
- 站内换乘不会错误生成中途出站。
- 只保留决策点，不每 100 米生成锚点。
- 长步行超过阈值自动插入安心确认点。
- 相邻锚点距离过近时自动缩小到达半径。
- 终点锚点使用百度路线实际落点，避免 POI 中心点偏移。

支持的决策点：

```text
START
STRAIGHT
LEFT
RIGHT
BUS_ON
BUS_OFF
SUBWAY_IN
SUBWAY_OUT
TRANSFER
DESTINATION
```

### 3.4 路线审核

已完成：

- 路线审核页展示完整决策点。
- 展示每步类型、风险等级、审核状态。
- 展示阻断发布原因。
- 家属可补照片、录音、生成 TTS、确认步骤。
- 家属可一键生成《真实路线采集清单》。
- 采集清单顶部只显示摘要和任务数量，具体采集建议按当前步骤展示，避免审核页被长清单撑开。
- 点“下一条”时，只显示下一步相关的拍照、地标、真人录音和发布前检查建议。
- 审核页已拆成“审核步骤 / 路线工具”两个工作区：
  - “审核步骤”只保留当前步骤、上一条、下一条、确认和采集建议。
  - “路线工具”集中放路线复盘、模拟执行、AI 语音、批量 TTS、采集清单摘要和完整路线。
  - 生成采集清单后自动回到“审核步骤”，避免家属每次翻下一条都要划过长工具区。
- 审核页可进入“路线复盘”，查看真实使用后的问题锚点。
- 高风险步骤要求家属确认。
- 高风险步骤缺少家属实拍照片不能发布。
- AI 生成内容不能自动发布。
- 低置信度 AI 文案需要家属确认。
- 未审核路线发布会被服务端拦截。

### 3.5 语音系统

已完成五类语音：

```text
enterVoiceText
repeatVoiceText
nearVoiceText
arrivedVoiceText
offRouteVoiceText
```

对应五类音频：

```text
enterAudioUrl
repeatAudioUrl
nearAudioUrl
arrivedAudioUrl
offRouteAudioUrl
```

支持三种语音来源：

```text
CUSTOM  家属真人录音
TTS     腾讯云语音合成
SYSTEM  系统默认文案
```

播放优先级：

```text
CUSTOM > TTS > SYSTEM 文案
```

已完成：

- 每一步支持五类语音。
- 每类语音可独立录音、试听、重录、删除。
- 批量生成 TTS。
- 批量 TTS 不覆盖真人录音。
- 单段 TTS 失败不影响整条路线。
- 老人端自动播放进入、重复、接近、到达、偏航语音。
- 防止语音重叠。
- 接近、到达每步只自动播一次。
- 偏航恢复后允许再次播报偏航。

### 3.6 AI 能力

已完成：

- 阿里百炼 `qwen-plus` 接入。
- `/api/engine/routes/advise` 路线顾问接口。
- `/api/engine/routes/{route_id}/ai-generate-voices` 步骤语音文案生成接口。
- `/api/engine/routes/{route_id}/collection-plan` 真实路线采集清单接口。
- AI 只基于已有百度路线和 RouteStep 生成文案。
- AI 不能新增、删除或修改步骤。
- AI 不能修改坐标、类型、公交线路、站名、出口。
- AI 失败时保留原 SYSTEM 文案。
- AI 采集清单失败时返回规则兜底清单，不阻塞路线审核。
- 服务端校验 AI 输出。

已加安全拦截：

- 非 `nearVoice` 中不得提前提示左转、右转。
- 禁止编造公交线路、站点、地铁线路。
- 禁止编造未在上下文出现的视觉地标。
- 公交未停稳时不得提示老人站起来。
- 终点进入语音不得提前说已经到达。
- 每条语音限制长度。
- 不确定表达会标记需要家属审核。

### 3.6.1 AI 采集助手

已完成：

- 根据已生成的 `RouteStep`、`riskLevel`、`type`、照片状态、地标、真人录音、TTS、AI 置信度和历史行程结果生成采集建议。
- 输出《真实路线采集清单》：
  - 必须拍照
  - 建议拍照
  - 必须补地标
  - 建议真人录音
  - 发布前检查
  - 实地测试步骤
- 能区分 `MUST / SHOULD / OPTIONAL`。
- 每个拍照任务给出原因、拍摄指导和错误示例。
- 每个地标任务给出建议地标类型和示例文案。
- 每个真人录音任务给出建议录音脚本。
- 会参考历史 `FOUND / NOT_FOUND / HELP` 结果，把容易找不到或求助的点提升为重点采集对象。
- AI 调用失败时使用规则兜底：
  - 高风险点必须拍照
  - 起点和终点必须拍照
  - 转弯点建议或必须拍照
  - 高风险点建议真人录音
  - 家属陪同测试
- AI 采集清单只作为建议，不会自动通过审核、不会自动发布路线，也不会把 `OPTIONAL` 任务变成发布阻断。
- 审核页展示方式已优化为“顶部摘要 + 当前步骤任务”，不再在页面顶部展开完整长清单。

### 3.6.2 AI 失败分析

已完成：

- 新增 `/api/engine/routes/{route_id}/trip-analysis`。
- 读取路线、步骤和 `FOUND / NOT_FOUND / HELP` 统计。
- 对出现 `NOT_FOUND / HELP` 的问题锚点生成优化建议。
- AI 不可用时使用规则兜底：
  - 找不到多：建议重拍照片、补地标
  - 求助多：建议加真人录音、提高风险等级、缩短锚点距离
  - 高风险点有问题：建议强制重新审核
- AI 只输出建议，不修改路线、不自动发布，结果仍需家属确认。

### 3.6.3 AI 照片审核

已完成规则占位：

- 新增 `/api/engine/routes/{route_id}/steps/{step_id}/photo-review`。
- 家属上传照片后自动进行规则检查并显示结果。
- 当前阶段不接多模态模型，只做 rule-based 判断：
  - `AUTO` 或静态地图类图片提示不能用于真实出行
  - 文件过小提示可能不清晰
  - 公交步骤提示拍清站名、线路号和方向
  - 转弯步骤提示拍清路口和转向方向
  - 终点步骤提示拍实际入口
- 照片审核结果不改变现有发布规则。

### 3.7 路线执行引擎

已完成：

- 每 5 秒定位一次。
- 后台计算当前锚点距离。
- 接近锚点触发 `NEAR`。
- 到达锚点触发 `ARRIVED`。
- 偏离路线触发 `OFF_ROUTE`。
- 支持 polyline 走廊偏航检测。
- 定位异常连续确认后才提示。
- 到达需要观察到接近过程，避免一进入就误到达。
- 旧路线运行时保护：
  - 旧 SYSTEM/TTS 提前转弯文案会被安全文案替换。
  - 真人录音不会被自动覆盖。
  - 旧终点锚点明显偏离路线落点时，执行时使用路线实际落点。

## 4. 最近真实验收结果

最近一次真实链路测试：

- 路线：上海市虹口区人民政府 → 复兴公园
- 百度路线距离：约 8.8 km
- 生成锚点：11 个
- 生成 AI 五类文案：11 / 11
- 生成 TTS 音频：55 / 55
- 模拟 GPS 跟踪锚点：9 / 9 可到达
- 偏航检测：正常
- 未审核发布：被正确拦截
- 微信开发者工具预览：通过

本轮发现并修复：

1. AI 提前说“现在左转 / 右转”。
2. AI 编造视觉地标。
3. 公交下车语音提示老人提前站起来。
4. 终点进入语音提前说已经到达。

## 5. 当前主要风险

### 5.1 真实照片风险

目前自动测试使用的是静态地图或占位，不是家属实拍照片。

真实使用前必须：

- 家属按路线实地走一遍。
- 每个关键锚点拍面向行进方向的照片。
- 高风险点必须拍清楚红绿灯、斑马线、站牌、入口等。
- 确认照片与语音一致。

### 5.2 AI 文案风险

AI 已有服务端安全拦截，但仍不能替代家属。

原因：

- AI 不知道现场施工、围挡、临时改道。
- AI 可能生成看似自然但现场不可识别的描述。
- AI 无法判断老人是否认识某个地标。

所以发布规则必须继续保留人工审核。

### 5.3 GPS 风险

GPS 只能辅助判断。

风险包括：

- 高楼、地铁口、公交站附近漂移。
- 锚点距离过近时可能连续触发。
- 老人实际步速差异很大。
- 公交和地铁场景不能只靠 GPS。

因此当前规则是接近后提示，但不自动切换下一步。

### 5.4 部署风险

本地 `127.0.0.1` 只适合开发者工具。

真机正式使用必须：

- 部署 HTTPS 公网服务。
- 配置微信 request 合法域名。
- 配置微信 uploadFile 合法域名。
- 服务端数据目录持久化。
- 小程序中不能出现百度、腾讯云、百炼长期密钥。

### 5.5 地铁场景风险

当前地铁只适合基础解析和演示。

暂不建议把以下场景作为 MVP 主打：

- 大型枢纽站。
- 站厅内多出口。
- 复杂换乘。
- 地下无 GPS 的长距离站内导航。
